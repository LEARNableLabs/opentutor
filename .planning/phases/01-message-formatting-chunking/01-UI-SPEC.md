# Phase 1 — UI Design Contract

> Visual and interaction contract for Telegram message delivery. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Note:** This phase delivers UI through Telegram Bot API (text messages with HTML formatting), not traditional web/mobile components. Design system = message formatting patterns.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Telegram Bot API native) |
| Preset | not applicable |
| Component library | Telegram Bot API HTML subset |
| Icon library | UTF-8 emoji (native) |
| Font | System default (Telegram client-controlled) |

**Rationale:** Telegram Bot API delivers messages as HTML-formatted text. No custom design system or component library needed. Typography, layout, and rendering are controlled by the Telegram client (iOS/Android app), not by this project.

---

## Spacing Scale

Declared values for message structure (character-based, not pixels):

| Token | Value | Usage |
|-------|-------|-------|
| line-break | 1 newline (`\n`) | Between paragraphs within a message |
| section-break | 2 newlines (`\n\n`) | Between semantic sections (concept → example) |
| message-gap | N/A (Telegram-controlled) | Between sequential messages (agent sends 3-4 messages, Telegram renders with native spacing) |

Exceptions: Code blocks use `<pre>` formatting with monospace spacing (Telegram-controlled).

**Rationale:** No pixel-based spacing scale. Message layout is controlled by Telegram client. Agent controls only newline placement within messages.

---

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | System default | 400 (regular) | Telegram default (~1.4) |
| Bold emphasis | System default | 700 (bold via `<b>` tag) | Telegram default (~1.4) |
| Code inline | Monospace (Telegram default) | 400 (regular via `<code>` tag) | Telegram default (~1.2) |
| Code block | Monospace (Telegram default) | 400 (regular via `<pre>` tag) | Telegram default (~1.2) |

**Rationale:** Typography is fully controlled by Telegram client. Agent can only apply semantic HTML tags (`<b>`, `<i>`, `<code>`, `<pre>`). Font family, size, and line-height are Telegram defaults (iOS uses San Francisco, Android uses Roboto).

**Font sizes (declared for copywriting guidance only):**
- Body: ~16px equivalent on mobile (Telegram default)
- No custom heading sizes — agent uses **bold** for emphasis within body text

**Font weights:**
- Regular (400): Default body text
- Bold (700): Emphasis via `<b>` or `<strong>` tags

**Line heights:**
- Body: ~1.4 (Telegram default for readability on mobile)
- Code: ~1.2 (Telegram default for monospace)

## Color

Color is controlled by Telegram client theme (light/dark mode). Agent cannot set custom colors.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | User's Telegram background | Message background |
| Secondary (30%) | User's Telegram bubble color | Message bubble (incoming messages from bot) |
| Accent (10%) | Emoji color (platform-native) | Structural anchors: 📖🧠💡✏️ |
| Text | User's Telegram text color | Message body (auto-adjusts for light/dark mode) |

Accent reserved for: Emoji anchors only (📖 Title/progress, 🧠 Core concept, 💡 Example, ✏️ Exercise). No other accent colors available.

**Rationale:** Telegram Bot API does not support custom text colors or background colors. All color is theme-controlled by user's Telegram client settings. Agent must design for both light and dark modes simultaneously.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | "Reply with your answer" (exercise prompts) |
| Empty state heading | Not applicable (no empty states in message delivery) |
| Empty state body | Not applicable |
| Error state | "Hmm, that doesn't look right. Try again?" (incorrect exercise answer) |
| Engagement prompt | "What do you think?" / "Give it a try" / "Tomorrow we'll see why this matters" (final message hook) |

**Message-specific copy patterns:**

| Pattern | Template | Example |
|---------|----------|---------|
| Lesson hook (📖) | "📖 Day {N}/{total} · {Topic} — {callback to yesterday}. Today: {teaser}?" | "📖 Day 5/34 · Eigenvalues — Yesterday we saw how matrices transform space. Today: what stays fixed?" |
| Concept intro (🧠) | "🧠 {definition}. {elaboration 1-2 sentences}." | "🧠 An eigenvalue λ tells you how much an eigenvector gets stretched. When a matrix multiplies a vector, most vectors change direction. But eigenvectors only stretch or shrink — they keep pointing the same way." |
| Example intro (💡) | "💡 {analogy or real-world example}." | "💡 Think of a door hinge. When you push the door, every point moves except the hinge. The hinge is an eigenvector with λ=1." |
| Exercise prompt (✏️) | "✏️ {problem statement} — {instruction}." | "✏️ Matrix A = [[2,1],[0,3]] — What are its eigenvalues? Reply with your answer." |

**Tone:** Conversational, Socratic, concrete examples over abstract definitions. Casual but not cutesy. (Source: `workspace/SOUL.md`)

## Message Formatting Contract

### HTML Tags (Telegram Bot API subset)

| Tag | Usage | Example |
|-----|-------|---------|
| `<b>`, `<strong>` | Key terms, definitions | "An <b>eigenvalue</b> λ tells you..." |
| `<i>`, `<em>` | Emphasis, variables | "most vectors change <i>direction</i>" |
| `<code>` | Inline code, math symbols | "If <code>Av = λv</code>, then..." |
| `<pre>` | Code blocks | `<pre>def eigenvalues(A):\n    return np.linalg.eigvals(A)</pre>` |
| `<pre><code class="language-{lang}">` | Syntax-highlighted code | `<pre><code class="language-python">...</code></pre>` |
| `<a href="...">` | Links | `<a href="https://example.com">Watch this video</a>` |
| `<u>`, `<ins>` | Underline (use sparingly) | Avoid unless semantically necessary |
| `<s>`, `<strike>`, `<del>` | Strikethrough (corrections) | "~~This is wrong~~ Actually..." |
| `<span class="tg-spoiler">` | Hidden text (reveals on tap) | `<span class="tg-spoiler">Answer: λ = 2, 3</span>` |

**Not allowed:** `<h1>`, `<p>`, `<div>`, custom CSS, custom colors. Telegram strips unsupported tags.

### Character Limits

| Limit | Value | Enforcement |
|-------|-------|-------------|
| Message length | 4,096 characters (hard limit) | Agent must chunk before hitting limit |
| Word count soft cap | 150 words per message | Agent aims for ~150, can exceed to 200 if splitting mid-thought hurts clarity (D-02) |
| Minimum messages | 2 per lesson | Even simple formats ("just a question") require at least 2 messages (D-03) |

### Emoji Anchor System

**4 structural anchors (D-04):**

| Emoji | Role | Placement | Usage |
|-------|------|-----------|-------|
| 📖 | Title/progress | First message of lesson | "📖 Day 5/34 · Eigenvalues — ..." |
| 🧠 | Core concept | Concept explanation message | "🧠 An eigenvalue λ tells you..." |
| 💡 | Example | Example/analogy message | "💡 Think of a door hinge..." |
| ✏️ | Exercise | Exercise/engagement prompt message | "✏️ Matrix A = [[2,1],[0,3]] — ..." |

**Rules:**
- One emoji anchor per message (placed at message start, before text)
- No decorative emoji (no 🎉✨🔥 for personality)
- Anchors are structural markers, not optional flourishes

**Replaced legacy anchors (from `openclaw/SOUL.md`):**
- ❌ 📌 Title → replaced by 📖
- ❌ 📚 References → removed (references go in final message without dedicated emoji)
- ❌ ⚡ Key distinction → merged into 🧠 (concept already covers distinctions)
- ❌ 🔬 Example → replaced by 💡

## Message Sequence Patterns

### Mini-Lesson (4 messages, most common)

**Source:** User-approved preview (01-CONTEXT.md, D-07)

```
Message 1 (📖 Hook):
📖 Day {N}/{total} · {Topic} — {callback to yesterday}. Today: {teaser}?

Message 2 (🧠 Concept):
🧠 {definition}.

{elaboration 1-2 sentences}.

Message 3 (💡 Example):
💡 {analogy or real-world example}.

{elaboration}.

Message 4 (✏️ Exercise):
✏️ {problem statement} — {instruction}.
```

**Example:**
```
📖 Day 5/34 · Eigenvalues — Yesterday we saw how matrices transform space. Today: what stays fixed?

🧠 An eigenvalue λ tells you how much an eigenvector gets stretched.

If Av = λv, then v is an eigenvector and λ is its eigenvalue.

When a matrix multiplies a vector, most vectors change direction. But eigenvectors only stretch or shrink — they keep pointing the same way.

💡 Think of a door hinge.

When you push the door, every point on the door moves — except the hinge. The hinge is an eigenvector with λ=1 (doesn't stretch at all).

✏️ Matrix A = [[2,1],[0,3]] — What are its eigenvalues?

Reply with your answer.
```

### Question Format (2 messages minimum)

**Pattern:** Hook → Question

```
Message 1 (📖 Hook):
📖 Day {N}/{total} — Quick one today...

Message 2 (🧠 Question):
🧠 Think about this: {thought-provoking question}?
```

### Resource Drop (3 messages)

**Pattern:** Hook → Resource → Follow-up

```
Message 1 (📖 Hook):
📖 Day {N}/{total} — Time for a different perspective...

Message 2 (💡 Resource):
💡 {resource description}: {link}. {Why it's valuable}.

Message 3 (✏️ Follow-up):
✏️ After {engaging with resource}, tell me: {reflection question}?
```

### Teach-Back (3 messages)

**Pattern:** Hook → Prompt → Scaffolding

```
Message 1 (📖 Hook):
📖 Day {N}/{total} — Let's see if you can explain this...

Message 2 (✏️ Prompt):
✏️ Explain {concept} in your own words. Pretend you're teaching it to someone who's never heard of it.

Message 3 (🧠 Scaffolding):
🧠 Hint: Start with {anchor point}. Then connect it to {example}.
```

### Challenge (3 messages)

**Pattern:** Hook → Challenge → Escape route

```
Message 1 (📖 Hook):
📖 Day {N}/{total} — Ready for a tricky one?

Message 2 (✏️ Challenge):
✏️ {hard problem statement}. Take your time.

Message 3 (💡 Hint):
💡 Stuck? Think about {hint}. Or reply "skip" and I'll show you the solution.
```

**Constraint (D-08):** Every final message must end with an engagement prompt (exercise, question, or teaser). No lessons end with "That's it for today!"

## Chunking Decision Tree

**When generating a lesson (source: RESEARCH.md, locked decisions D-01, D-02, D-03):**

1. **Identify pedagogical sections** (hook, concept, example, exercise)
2. **For each section:**
   - Draft content
   - Check word count
   - **If < 150 words:** One message
   - **If 150-200 words:** Assess: can split at paragraph boundary? If yes, split. If no, keep as one message (soft cap allows exceeding for clarity).
   - **If > 200 words:** Split at semantic boundary (after definition, after example setup, between paragraphs)
3. **Assign emoji anchor** based on section type (📖🧠💡✏️)
4. **Ensure final message has engagement prompt** (D-08)
5. **Ensure total message count ≥ 2** (D-03)

**Exception:** Code blocks kept whole even if > 150 words. Splitting code breaks syntax. Code blocks are visually distinct (monospace, `<pre>` formatting) — not "wall of text."

## Validation Checklist

**Manual testing required (no automated framework):**

- [ ] **Message count:** Deliver sample mini-lesson via OpenClaw. Verify 3-4 messages received (not 1 wall of text).
- [ ] **Word count:** Count words in each message. Verify ≤ 150 words per message (or ≤ 200 if section requires clarity).
- [ ] **Emoji anchors:** Verify exactly 4 emoji types used (📖🧠💡✏️). No decorative emoji (🎉✨🔥).
- [ ] **HTML formatting:** Verify bold (`<b>`), italic (`<i>`), code (`<code>`), code blocks (`<pre>`) render correctly on iPhone and Android Telegram apps.
- [ ] **Engagement prompt:** Verify final message ends with question, exercise, or teaser (not "That's it for today!").
- [ ] **Character limit:** Verify no message exceeds 4,096 characters (Telegram API hard limit).
- [ ] **Light/dark mode:** Test on both Telegram light and dark themes. Verify readability in both.
- [ ] **Semantic splitting:** Verify messages split at concept/example/exercise boundaries, not mid-sentence.

**Platform:** iPhone Telegram app + Android Telegram app (mobile-first validation)
